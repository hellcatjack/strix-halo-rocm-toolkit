# v0.3.2

`v0.3.2` 修复 stable 镜像在 Docker containerd image store 上下载完成后被错误判定为 config digest 不匹配的问题。正式容器镜像、ROCm 7.2.1、Python 3.12、PyTorch 2.9.1、stable release ID `0.2.0` 及 exact digest 均不变。

## 根因与修复

经典 Docker image store 的 `docker image inspect .Id` 是 image config digest；containerd image store 将该字段暴露为本地 target，也就是本次固定引用的 manifest digest。`v0.3.1` 只接受前一种语义，因此把正确下载的镜像误判为身份漂移。

`v0.3.2` 只接受以下两种本地 ID：

- release 清单中的 exact config digest；
- release 清单中的 exact manifest digest。

当本地 ID 是 manifest digest 时，工具使用空 Docker 凭据目录匿名读取该固定 manifest 的 config descriptor，并继续验证 exact RepoDigest、OCI labels 和镜像内锁文件摘要。随机 ID、descriptor 漂移、标签漂移或内嵌文件漂移仍然失败关闭；不会回退到 mutable tag，也不会把身份失败降级为本地构建。

项目初始化、父镜像友好标签、Doctor/repair、缺失层估算和本地构建也采用同一套双后端语义。项目配置中的 `base_manifest_digest`/`base_digest` 保存完整验证身份对，`base_image` 保存 Docker 可寻址的其中一个本地不可变 ID；已有项目和后续构建会重新核对三者。已完整下载的 containerd 镜像不会被重复计为 8.5 GiB 缺失层。

## 从 v0.3.1 恢复

```bash
cd "$HOME/src/strix-halo-rocm-toolkit"
git fetch --tags origin
git switch --detach v0.3.2

PROJECT="$HOME/ai-projects/video-lab"
./install.sh --mode full \
  --project-dir "$PROJECT" \
  --project-name video-lab \
  --image-source pull
```

不要删除项目状态、镜像或 `/var/lib/docker`。已完成下载的层由 Docker 内容存储复用；安装器会重验只读内核/宿主关卡，跳过已经完成的写入阶段，然后从 `IMAGE_PULL_OR_BUILD` 继续。
