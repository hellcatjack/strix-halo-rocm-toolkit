# AMD AI Container Platform Implementation Plan Index

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deliver the approved Ryzen AI Max+ 395 ROCm 7.2.1 container platform through four bounded, testable implementation plans.

**Architecture:** A Python 3.12 standard-library management CLI owns host inspection, image orchestration, project configuration, and release reports; thin executable wrappers expose the approved command names. Ubuntu host changes, OCI image construction, project runtime policy, and hardware qualification remain separate modules connected through versioned JSON reports and immutable image/profile identifiers.

**Tech Stack:** Python 3.12, pytest, Bash wrappers, Docker Engine/BuildKit, Ubuntu 24.04, ROCm 7.2.1, `uv` 0.11.28, PyTorch 2.9.1, HIP, Triton 3.5.1.

## Locked Cross-Plan Interfaces

### Command surface

```text
bin/host-preflight
bin/host-prepare
bin/host-verify
bin/image-build
bin/project-init
bin/project-run
bin/container-check
```

All wrappers dispatch into `python3 -m amd_ai.cli`. They return nonzero on blocking findings and support `--json PATH` where a persistent report is useful.

### Stable identifiers

```text
ROCm/Python image: rocm-python:7.2.1-py3.12
Verified Torch image: rocm-pytorch:7.2.1-py3.12-torch2.9.1
Verified profile ID: rocm-7.2.1-py3.12-torch-2.9.1
Target architecture: linux/amd64
Target GPU architecture prefix: gfx1151
```

### Report envelope

Every JSON report uses this top-level shape:

```json
{
  "schema_version": 1,
  "command": "host-preflight",
  "generated_at": "2026-07-09T12:00:00Z",
  "status": "pass",
  "facts": {},
  "findings": []
}
```

`status` is one of `pass`, `change-required`, `reboot-required`, `unverified`, or `blocked`. Each finding contains `code`, `severity`, `summary`, `evidence`, and `remediation`.

### Profile contract

Torch profile files use strict `KEY=VALUE` syntax without shell evaluation. Required keys are:

```text
PROFILE_ID
PROFILE_STATUS
ROCM_VERSION
PYTHON_ABI
PLATFORM
TORCH_VERSION
TORCH_URL
TORCH_SHA256
TORCHVISION_VERSION
TORCHVISION_URL
TORCHVISION_SHA256
TORCHAUDIO_VERSION
TORCHAUDIO_URL
TORCHAUDIO_SHA256
TRITON_VERSION
TRITON_URL
TRITON_SHA256
```

The parser rejects duplicate keys, unknown keys, shell substitutions, missing hashes, non-HTTPS URLs, and any status other than `verified` or `experimental`. Only the repository-owned stable profile may declare `verified`.

### Project contract

Each project owns an `amd-ai-project.toml`, a project Dockerfile, a dependency input file, and a resolved lock file. The project config may declare arbitrary bind mounts with explicit read-only/read-write mode, but no model or Hugging Face mount is generated by default.

### Image labels

```text
org.amd-ai.profile.id
org.amd-ai.profile.status
org.amd-ai.rocm.version
org.amd-ai.python.version
org.amd-ai.torch.version
org.opencontainers.image.revision
org.opencontainers.image.source
```

`project-run` reads these labels before constructing `docker run`. An `experimental` profile requires `ALLOW_UNVERIFIED=1`; a missing label is treated as `experimental`.

## Source Baseline

- AMD requires the 6.14-1018 OEM kernel or newer for this Ryzen ROCm line and documents the inbox-driver approach.
- AMD's ROCm 7.2.1 Ryzen page publishes the exact CPython 3.12 wheel filenames for Torch 2.9.1, TorchVision 0.24.0, TorchAudio 2.9.0 and Triton 3.5.1.
- AMD documents minimum 0.5 GiB BIOS dedicated VRAM and a larger TTM allocation managed by `amd-ttm`.

Primary references are retained in the approved design specification and repeated where a child plan depends on exact commands.
